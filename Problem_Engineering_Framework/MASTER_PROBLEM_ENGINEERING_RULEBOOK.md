# PROBLEM ENGINEERING RULEBOOK

## Version 1.0

### Gate DA Platform

---

# 1. PURPOSE

This document defines the official standards, methodology, workflows, and quality requirements for creating, reviewing, maintaining, and evolving problems on the platform.

The objective is not to build a question bank.

The objective is to build a structured problem-solving ecosystem that develops:

* Conceptual mastery
* Mathematical maturity
* Multi-step reasoning
* Algorithmic thinking
* Research-oriented problem solving
* Interview readiness
* Olympiad-style creativity

---

# 2. CORE PHILOSOPHY

Every problem must satisfy at least one of the following:

* Teaches a concept
* Reveals a hidden observation
* Builds intuition
* Connects multiple ideas
* Demonstrates a useful technique
* Tests reasoning under constraints
* Encourages abstraction

Problems should never exist solely to test formula memorization.

---

# 3. PROBLEM CREATION PRINCIPLES

## Rule 1

Never generate directly from a chapter.

Wrong:

Chapter → Problem

Correct:

Chapter
→ Topic
→ Subtopic
→ Concept
→ Pattern
→ Difficulty
→ Reasoning Type
→ Problem

---

## Rule 2

Every problem must have a primary learning objective.

Examples:

* Conditional reasoning
* Bayesian updating
* Symmetry observation
* Counting argument
* Optimization insight

---

## Rule 3

Every problem must belong to a Pattern ID.

Examples:

SE_P001
SE_P002
CP_P004
RV_P012

No problem can exist without a pattern.

---

## Rule 4

Every problem must be independently valuable.

Removing the answer should still leave a meaningful learning experience.

---

# 4. PROBLEM DIFFICULTY FRAMEWORK

## L1 Foundation

Purpose:

Concept recognition

Expected Skills:

* Definitions
* Direct interpretation
* Simple application

Time:

30 sec – 2 min

---

## L2 Standard GATE

Purpose:

Concept application

Expected Skills:

* Formula selection
* Basic reasoning
* Standard derivations

Time:

2 – 5 min

---

## L3 Analytical

Purpose:

Multi-step reasoning

Expected Skills:

* Hidden observations
* Combining concepts
* Event transformations

Time:

5 – 10 min

---

## L4 IIT/IISc

Purpose:

Mathematical maturity

Expected Skills:

* Novel formulations
* Advanced abstraction
* Non-obvious solutions

Time:

10 – 20 min

---

## L5 Olympiad

Purpose:

Creativity

Expected Skills:

* Symmetry
* Invariance
* Construction
* Elegant observations

Time:

20 – 40 min

---

## L6 Research

Purpose:

Generalization

Expected Skills:

* Modeling
* Assumption analysis
* Open-ended reasoning

Time:

Variable

---

# 5. REASONING TYPE FRAMEWORK

Every problem must contain one or more reasoning tags.

Allowed tags:

* Computational
* Analytical
* Geometric
* Visual
* Combinatorial
* Probabilistic
* Statistical
* Optimization
* Algorithmic
* Graph Based
* Bayesian
* Inference
* Simulation
* Modeling
* Proof

---

# 6. VISUALIZATION POLICY

Whenever visualization significantly improves understanding, it must be used.

Preferred visualization types:

* Venn Diagrams
* Probability Trees
* Sample Space Grids
* Distribution Curves
* CDF Plots
* Geometric Probability Diagrams
* Graph Structures
* Markov Chains
* Random Walk Diagrams
* Scatter Plots
* Heat Maps
* State Machines

Visualization should never be decorative.

Visualization must improve reasoning.

---

# 7. ORIGINALITY RULES

Forbidden:

* Direct PYQ copy
* Direct textbook copy
* Trivial numerical modifications
* Repeated templates

Allowed:

* Concept extraction
* Structural transformation
* Pattern recombination
* Context replacement
* Difficulty escalation

---

# 8. PROBLEM GENERATION WORKFLOW

Mandatory sequence:

Concept
→ Pattern
→ Reasoning Type
→ Difficulty
→ Visualization
→ Exam Style
→ Problem
→ Solution
→ Validation

Skipping steps is prohibited.

---

# 9. PROBLEM PATTERN ENGINEERING

Each subtopic must define:

* Concept Patterns
* Standard Patterns
* Advanced Patterns
* Olympiad Patterns
* Interview Patterns
* Research Patterns
* Fusion Patterns

Problems are generated from patterns.

Never directly from topics.

---

# 10. FUSION PROBLEM POLICY

The highest quality problems usually combine concepts.

Examples:

Probability + Geometry

Probability + Graph Theory

Probability + Optimization

Statistics + Linear Algebra

Statistics + Machine Learning

Bayesian Inference + Optimization

Random Variables + Algorithms

Random Walks + Markov Chains

Information Theory + Probability

---

# 11. EXAM STYLE FRAMEWORK

Every problem must belong to at least one category.

Categories:

* GATE
* IIT
* IISc
* ISI
* CMI
* TIFR
* Olympiad
* Interview
* Research

---

# 12. SOLUTION ENGINEERING STANDARD

Every solution must contain:

1. Overview
2. Key Observation
3. Visualization
4. Derivation
5. Shortcut Insight
6. Common Trap
7. Final Answer

Solutions should teach.

Solutions should not merely verify.

---

# 13. COMMON TRAP REQUIREMENT

Every analytical problem must identify:

* Likely mistake
* False intuition
* Common misconception

This improves learning quality.

---

# 14. QUALITY RUBRIC

Each problem receives scores:

Conceptual Depth: 10

Originality: 10

Reasoning Value: 10

Clarity: 10

Visualization Quality: 10

Exam Relevance: 10

Learning Value: 10

Total: 70

Minimum approval score:

55

---

# 15. OLYMPIAD STANDARDS

Olympiad problems should emphasize:

* Symmetry
* Invariance
* Construction
* Extremal reasoning
* Elegant observations

Avoid brute-force calculations.

---

# 16. RESEARCH STANDARDS

Research problems should emphasize:

* Modeling
* Generalization
* Sensitivity analysis
* Interpretation
* Open assumptions

The goal is exploration.

Not computation.

---

# 17. INTERVIEW STANDARDS

Interview problems should test:

* Thinking process
* Assumption discovery
* Communication
* Approximation
* Modeling

The final answer is often less important than the reasoning.

---

# 18. AI GENERATION POLICY

AI must receive:

Subject
Chapter
Topic
Subtopic
Concept
Pattern ID
Difficulty
Reasoning Type
Visualization Type
Exam Style

AI should never receive only:

"Generate a probability question"

This produces low-quality content.

---

# 19. PROBLEM BANK DISTRIBUTION TARGET

Recommended distribution:

L1 Foundation: 20%

L2 Standard GATE: 30%

L3 Analytical: 25%

L4 IIT/IISc: 15%

L5 Olympiad: 7%

L6 Research: 3%

---

# 20. GOLDEN RULE

A problem is approved only if:

It teaches something valuable.

If solving the problem does not improve the student’s understanding, intuition, reasoning ability, or mathematical maturity, the problem should not exist on the platform.
